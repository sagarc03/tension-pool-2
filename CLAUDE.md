## Project Overview

Tension Pool 2 is a Foundry VTT V13+ module — a reimplementation and evolution of the Angry GM's Tension Pool concept. Spiritual successor to [SDoehren/tension-pool](https://github.com/SDoehren/tension-pool).

## Tech Stack

TypeScript, Vite (library mode), Vitest, pnpm, plain CSS. Foundry types via `@league-of-foundry-developers/foundry-vtt-types`.

## Commands

| Command          | Description                                      |
|------------------|--------------------------------------------------|
| `pnpm dev`       | Build + watch (outputs to `build/`)              |
| `pnpm build`     | One-shot build to `build/`                       |
| `pnpm test`      | Run Vitest                                       |
| `pnpm test:watch`| Run Vitest in watch mode                         |
| `pnpm release`   | Build + generate `module.json` + zip into `dist/`|

## Architecture

- `src/scripts/module.ts` — Entry point. Registers Foundry hooks, settings, UI.
- `src/styles/module.css` — Module styles. Imported by module.ts so Vite processes it.
- `src/templates/` — Handlebars templates. Copied to `build/` by Vite plugin (not bundled).
- `build/` — Compiled module output. Mounted into Foundry via Docker for local dev.
- `dist/` — Release artifacts (`module.json` + `module.zip`). Generated by `pnpm release`.
- `scripts/release.mjs` — Generates `module.json` from `package.json` and zips `build/`.

## Metadata

`package.json` is the single source of truth. Foundry-specific fields (`compatibility`, `socket`, etc.) live under the `"foundry"` key. No `module.json` in source — it's generated at release time.

## Testing

Core logic should be pure TypeScript with no Foundry dependencies — testable directly with Vitest. Foundry integration code (hooks, settings, UI) can use lightweight mocks when needed.

## Local Dev with Docker

`docker compose up` runs Foundry VTT with `build/` symlinked as the module. Requires `.env` with `FOUNDRY_USERNAME` and `FOUNDRY_PASSWORD` (see `.env.example`). An init container runs as root to fix `/data/Data` permissions before Foundry starts. Named volumes cache the Foundry installation and data.

## Release

Push a `v*` tag to trigger the GitHub Actions workflow, which runs tests, builds release artifacts, and creates a GitHub Release with `module.json` and `module.zip` attached.
