# Contributing to Tension Pool 2

## Prerequisites

- [Node.js](https://nodejs.org/) v22+
- [pnpm](https://pnpm.io/) v10+
- [Docker](https://www.docker.com/) (for running Foundry VTT locally)
- A [Foundry VTT](https://foundryvtt.com/) license

## Setup

1. Clone the repository:

   ```bash
   git clone https://github.com/sagarc03/tension-pool-2.git
   cd tension-pool-2
   ```

2. Install dependencies:

   ```bash
   pnpm install
   ```

3. Create a `.env` file from the template:

   ```bash
   cp .env.example .env
   ```

4. Fill in your Foundry credentials in `.env`:

   ```
   FOUNDRY_USERNAME=your-username
   FOUNDRY_PASSWORD=your-password
   ```

   These are your [Foundry VTT website](https://foundryvtt.com/) account credentials.

## Local Development

Run two processes — one for the build watcher, one for Foundry:

1. Start the Vite build watcher:

   ```bash
   pnpm dev
   ```

   This compiles TypeScript and CSS into `build/` and rebuilds on file changes. Note: changes to `src/lang/*.json` and `src/templates/*.hbs` require a full rebuild (`pnpm build`) since they are copied by a Vite plugin in `closeBundle`, not watched by HMR.

2. In a separate terminal, start Foundry VTT:

   ```bash
   docker compose up
   ```

   Foundry will be available at [http://localhost:30000](http://localhost:30000). The first run downloads Foundry (cached in a Docker volume for subsequent runs).

3. Enable the "Tension Pool 2" module in your Foundry world's module settings.

4. After making code changes, reload Foundry in your browser to pick them up.

## Project Structure

```
src/
  scripts/
    module.ts            — Entry point. Registers hooks, settings, and UI
    tension-pool-app.ts  — ApplicationV2 class for the floating pool UI
    tension-die.ts       — Custom TensionDie, roll logic, Dice So Nice integration
    quench.ts            — End-to-end tests (dev only, runs inside Foundry)
  styles/
    module.css           — All module styles
  templates/
    pool.hbs             — Handlebars template for the pool UI
  lang/
    en.json              — English localization strings
test/
  tension-die.test.ts    — Vitest unit tests for roll logic
  example.test.ts        — Smoke test for test setup
assets/                  — Screenshots for README
build/                   — Compiled output (mounted into Foundry via Docker)
dist/                    — Release artifacts (generated by pnpm release)
scripts/
  release.mjs            — Generates module.json from package.json and zips build/
```

## Architecture

### Key Concepts

- **`TensionPoolApp`** — A frameless `ApplicationV2` with `HandlebarsApplicationMixin`. Uses `window: { frame: false, positioned: false }` to render as a bare floating element. Positioned next to the macro hotbar using `offsetLeft`/`offsetTop` calculations that account for `--ui-scale`.

- **`TensionDie`** — Custom `Die` subclass with denomination `"t"`. Reads the `diceSize` setting to determine face count (d4-d20). The `total` getter counts 1s instead of summing, since a 1 is what triggers a complication.

- **Settings as state** — The current dice count (`diceCount`) is stored as a world-level setting. This means changes propagate to all clients via Foundry's built-in setting sync. The `onChange` callback re-renders the UI.

- **`root: true` template behavior** — When the template has a single root element (e.g., non-GM view with only `.tp-icons`), that element replaces the `#tension-pool` app element. CSS accounts for this with `#tension-pool:not(:has(.tp-icons))`.

- **Client-side icon theming** — Chat messages use `data-tp-icon` attributes instead of hardcoded FA classes. A `renderChatMessage` hook resolves these to the client's chosen icon theme.

### Hooks

The module fires two custom hooks after a roll:

- `tensionPoolRolled` — Every roll, with `{ diceCount, results, hasComplication, complicationCount }`
- `tensionPoolComplication` — Only on complications, same payload

### Dice So Nice Integration

When DSN is active, rolls use the custom `dt` die type for 3D rendering. The preset is registered in the `diceSoNiceReady` hook with a "!" label on face 1. When DSN is not present, standard Foundry dice (`d6`, `d8`, etc.) are used instead.

## Running Tests

### Unit Tests (Vitest)

```bash
pnpm test          # Single run
pnpm test:watch    # Watch mode
```

Unit tests are in `test/` and mock Foundry globals. They cover the core roll logic, complication detection, and hook firing.

### End-to-End Tests (Quench)

Quench tests run inside a live Foundry instance. To use them:

1. Install the [Quench](https://foundryvtt.com/packages/quench) module in Foundry
2. Enable both "Tension Pool 2" and "Quench" in your world
3. Open the Quench test runner from the sidebar
4. Run the "Tension Pool 2" test batches

Quench tests cover:
- **Basic** — Module is active
- **Settings** — All settings are registered and accessible
- **UI & Controls** — DOM elements exist, GM buttons work, pool manipulation (add/remove/roll/clear), boundary conditions
- **Tension Die** — Custom die registration, roll evaluation, complication counting

Quench tests are only loaded in dev builds (`import.meta.env.DEV`) and excluded from release artifacts.

## Adding a New Setting

1. Register it in `src/scripts/module.ts` inside the `init` hook
2. Add localization keys to `src/lang/en.json`
3. If it affects the UI, add `onChange: () => { poolApp?.render({ force: true }); }`
4. Read it in the app via `(game as Game).settings!.get("tension-pool-2" as any, "key" as any)`

## Adding a New Icon Theme

1. Add the theme to `ICON_THEMES` in `src/scripts/tension-pool-app.ts`
2. Add a choice entry in the `iconTheme` setting registration in `src/scripts/module.ts`
3. Add the localization string to `src/lang/en.json`

## Releasing

Releases are automated via GitHub Actions. To create a release:

1. Update the version in `package.json`.
2. Commit the version bump.
3. Tag the commit and push:

   ```bash
   git tag v1.0.0
   git push origin main --tags
   ```

4. The GitHub Actions workflow will run tests, build the module, and create a GitHub Release with `module.json` and `module.zip` attached.
